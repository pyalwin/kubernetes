A container image should be a *build artifact*, not a *configuration file*. The
same image that runs in your laptop, staging, and production needs different
database URLs, feature flags, and credentials in each place. Baking those into
the image would force a rebuild per environment and tempt you to commit secrets
into source control. Kubernetes solves this by **externalizing config** into two
objects: **ConfigMaps** and **Secrets**.

## ConfigMaps: non-sensitive configuration

A **ConfigMap** holds non-sensitive key/value data: log levels, feature flags,
hostnames, whole config files. You create one with `kubectl create configmap` or
a YAML manifest, then inject its values into a Pod. There are two main ways to
inject:

- **As environment variables** — map keys become `env` entries the process reads
  at startup. Simple, but the process won't see changes without a restart.
- **As mounted files** — the ConfigMap is mounted as a volume, so each key
  becomes a file in a directory. This suits apps that read a config file, and
  mounted values *can* update live (with a short delay).

## Secrets: sensitive data

A **Secret** is structurally almost identical to a ConfigMap but is meant for
sensitive data: passwords, API tokens, TLS certificates. Two things to
understand clearly:

- Secret values are **base64-encoded, not encrypted** by default. Base64 is
  encoding, not security — anyone who can read the Secret object can trivially
  decode it. For real protection you enable **encryption at rest** in etcd and
  lock down access with RBAC.
- Kubernetes treats Secrets a bit more carefully than ConfigMaps: they can be
  kept in tmpfs (memory) when mounted, and access is more tightly controlled.

You inject Secrets the same two ways — as env vars or mounted files. Mounting as
files is generally preferred for sensitive data, since env vars are easier to
leak through logs or child processes.

## Why this separation matters

Keeping config and secrets out of the image gives you one immutable, tested
artifact promoted unchanged across environments — only the injected ConfigMap or
Secret differs. It also draws a clean line between *non-sensitive* settings
anyone can read and *sensitive* material that deserves tighter handling. When you
later study security and RBAC, that line is exactly where access controls attach.
