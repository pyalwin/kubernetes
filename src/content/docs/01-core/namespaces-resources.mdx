A single cluster is usually shared — by multiple teams, multiple environments, or
many apps. Two mechanisms keep that sharing orderly: **namespaces** carve the
cluster into logical partitions, and **resource requests and limits** govern how
much compute each workload may consume. Together they make multi-tenancy safe.

## Namespaces: logical partitions

A **namespace** is a virtual cluster inside your cluster. It provides a *scope
for names* — you can have a `web` Deployment in both `team-a` and `team-b`
without collision — and a boundary for organization, access control, and quotas.
Most objects (Pods, Services, Deployments, ConfigMaps) are namespaced; a few
(Nodes, PersistentVolumes, namespaces themselves) are cluster-wide.

Out of the box you get `default`, `kube-system` (control-plane components), and
`kube-public`. In practice you create namespaces per team or environment and
attach **RBAC** rules so each team only touches its own. DNS reflects this:
`web.team-a.svc.cluster.local` names the `web` Service in `team-a`.

## Requests and limits

By default a container can grab as much CPU and memory as the node has — a noisy
neighbor can starve everyone else. You prevent that with two numbers per
container:

- A **request** is what the container is *guaranteed*. The **scheduler** uses
  requests to decide which node has room: it only places a Pod where the sum of
  requests still fits. Requests are about *placement*.
- A **limit** is the *hard ceiling* enforced at runtime through Linux
  **cgroups**. Exceed a **memory** limit and the container is **OOM-killed**.
  Exceed a **CPU** limit and the container is simply **throttled** (slowed), not
  killed — an important asymmetry.

Setting both well is the core of healthy scheduling: requests keep nodes from
being oversubscribed, limits keep one workload from hogging a node.

## ResourceQuota and LimitRange

Two namespace-level controls build on this. A **ResourceQuota** caps the *total*
requests and limits (and object counts) a whole namespace may use — so `team-a`
can't consume the entire cluster. A **LimitRange** sets *default* and
min/max requests and limits for individual containers in a namespace, so Pods
created without explicit values still get sensible ones. Used together, they let
a platform team hand out a namespace with guardrails already in place.
