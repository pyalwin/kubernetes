A Kubernetes cluster is split into two kinds of machines that do very different
jobs. The **control plane** is the brain: it stores what you declared and runs
the loops that make reality match. The **worker nodes** are the muscle: they
actually run your containers. Understanding this split is the key that unlocks
everything else in Part 1.

## The control plane

The control plane is a small set of cooperating components. Click each one to
see its role:

<Diagram client:visible id="control-plane" title="The Kubernetes control plane" nodes={controlPlaneNodes} />

- **kube-apiserver** is the front door. It is the *only* component that talks to
  the database, and everything else — you, the controllers, the nodes — talks to
  the cluster through it. It validates and authorizes every request. This
  API-server-centric design is deliberate: there is one consistent gateway, so
  every actor sees the same state.
- **etcd** is the cluster's memory: a consistent, distributed key-value store
  holding every object you declared (`spec`) and every observed status
  (`status`). If etcd is the source of truth, losing it means losing the
  cluster, which is why it is backed up carefully.
- **kube-scheduler** watches for Pods that have no node yet and assigns each to a
  node, weighing free resources, affinity rules, taints, and tolerations.
- **kube-controller-manager** runs the reconciliation loops — the Node
  controller, ReplicaSet controller, Job controller, and many more — each
  watching its slice of state and nudging it toward desired.
- **cloud-controller-manager** isolates cloud-provider logic (provisioning load
  balancers, attaching disks, labeling nodes) so the core stays vendor-neutral.

## The worker node

Every worker node runs three things. The **kubelet** is the node agent: it
watches the API server for Pods assigned to its node and tells the container
runtime to start them, then reports health back. **kube-proxy** programs the
node's networking rules so traffic to a Service's virtual IP reaches a real Pod.
The **container runtime** (commonly **containerd**) is the engine that actually
pulls images and runs containers via the CRI interface.

## How a `kubectl apply` flows through

Say you run `kubectl apply -f deploy.yaml`. The request hits the **API server**,
which validates it and writes the desired object into **etcd**. The relevant
**controller** notices the new Deployment and creates a ReplicaSet, which
creates Pod objects. The **scheduler** sees those unscheduled Pods and assigns
each a node. Finally the **kubelet** on that node pulls the image and starts the
container. No single component did the whole job — each watched its piece and
reacted. That is reconciliation in action.
