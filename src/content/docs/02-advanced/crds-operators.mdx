Kubernetes ships with built-in kinds — Pod, Service, Deployment. But its real power is
that the API is **extensible**: you can teach the cluster entirely new object types and
new behavior. That's how tools like cert-manager, Prometheus, and database operators
feel like native parts of Kubernetes. Two ideas make it work: CRDs and the Operator
pattern.

## CustomResourceDefinitions — new kinds in the API

A **CustomResourceDefinition (CRD)** registers a brand-new resource kind with the API
server. Once you apply a CRD for, say, `Database`, the cluster understands it as a
first-class object: you can `kubectl get databases`, write one in YAML, store it in
etcd, and apply RBAC to it — exactly like a built-in resource.

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata: { name: databases.example.com }
spec:
  group: example.com
  names: { kind: Database, plural: databases }
  scope: Namespaced
```

On its own, a CRD is **inert** — it's just a typed record the API server will store and
validate. Creating a `Database` object doesn't actually create a database. Something has
to *act* on it. That something is a controller.

## The Operator pattern — CRD + controller

Recall the **control loop** from Part 0: every Kubernetes controller continuously
**observes** actual state, **compares** it to the declared desired state, and **acts** to
close the gap, forever. An **Operator** is exactly that idea applied to a custom resource.
An operator is:

> **a CustomResourceDefinition + a custom controller** that runs the reconcile loop for
> that resource's domain logic.

The controller **watches** `Database` objects. When you create one declaring "Postgres 16,
20Gi, 3 replicas," the controller reconciles reality toward it: it provisions a
StatefulSet, creates Services and Secrets, sets up backups. When you edit the object to
ask for 5 replicas, the loop notices the gap and scales up. When a pod dies, it heals.
The operator encodes the knowledge a human operator would otherwise apply by hand — how
to deploy, upgrade, back up, and recover *this specific software* — as code that runs
24/7.

## Why it matters

This is the same reconcile-loop principle that runs all of Kubernetes, now extended to
*your* domain. Real examples: the **Prometheus Operator** turns a `Prometheus` resource
into a fully wired monitoring stack; cert-manager turns a `Certificate` resource into
issued, auto-renewed TLS certs. Instead of bolting automation on *around* Kubernetes,
operators put your domain logic *inside* it — declarative, self-healing, and managed by
the same `kubectl apply` workflow as everything else. Custom resources plus a control
loop is the pattern that lets Kubernetes manage almost anything.
