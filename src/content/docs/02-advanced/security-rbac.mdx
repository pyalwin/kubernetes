A cluster's API server is the single front door to everything — create a pod, read a
Secret, delete a node. **RBAC** (Role-Based Access Control) decides who may do what.
Get it right and a compromised workload stays boxed in; get it wrong and one leaked
token owns the cluster.

## Verbs, resources, subjects

RBAC answers a simple question: can this **subject** perform this **verb** on this
**resource**? 

- **Verbs** are actions: `get`, `list`, `watch`, `create`, `update`, `delete`.
- **Resources** are object kinds: `pods`, `secrets`, `deployments`, `nodes`.
- **Subjects** are who's asking: users, groups, or ServiceAccounts.

A **Role** is a set of allowed verb-on-resource rules. A **RoleBinding** grants a Role
to subjects. Rules are purely additive — there are no "deny" rules — so a subject can
do exactly what some binding grants and nothing more.

## Role vs. ClusterRole

Scope is the key distinction:

- A **Role** + **RoleBinding** are **namespaced** — they grant permissions within one
  namespace. "Read pods in `team-a`."
- A **ClusterRole** + **ClusterRoleBinding** apply **cluster-wide** and to
  cluster-scoped resources like `nodes` and `namespaces`. "Read pods in *every*
  namespace," or "manage nodes."

A handy combination: bind a **ClusterRole** with a namespaced **RoleBinding** to reuse
a common permission set (e.g. "view") inside a single namespace.

## ServiceAccounts — identity for pods

Humans authenticate with certificates or OIDC, but **pods** need an identity too. That
identity is a **ServiceAccount**. A token for the pod's ServiceAccount is mounted into
the container, and the pod presents it when calling the API server. The rule of
**least privilege** follows directly: create a dedicated ServiceAccount per app and
bind it to a Role granting only what that app needs. Never let workloads run as
`cluster-admin` — if that pod is breached, so is the cluster.

```yaml
kind: RoleBinding
metadata: { name: read-pods, namespace: team-a }
subjects:
- kind: ServiceAccount
  name: log-reader
roleRef: { kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io }
```

## Pod security: securityContext and the restricted profile

RBAC controls API access; **securityContext** hardens the running container. Set it to
drop Linux capabilities, run as a non-root user, and forbid privilege escalation:

```yaml
securityContext:
  runAsNonRoot: true
  allowPrivilegeEscalation: false
  capabilities: { drop: ["ALL"] }
```

Kubernetes packages these into **Pod Security Standards** — `privileged`, `baseline`,
and the strict **`restricted`** profile, which enforces non-root, no privilege
escalation, and a tight capability set. Applying the `restricted` profile to a
namespace shrinks the blast radius of any container that gets compromised. Defense in
depth: RBAC at the API, securityContext at the kernel.
