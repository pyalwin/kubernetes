Kubernetes is, to a large degree, a networking system: its job is to let a
shifting set of containers find and talk to each other reliably. Three classic
ideas carry almost all of that weight, and the cluster builds higher-level
features on top of them.

## TCP/IP — how two endpoints talk

Every container gets an **IP address**, and processes communicate over
**TCP/IP**: data is split into packets, routed across the network, and
reassembled in order. A connection is identified by the pair of `IP:port` on
each end. When you later expose a Service "on port 80," this is the layer doing
the actual work.

## DNS — names instead of addresses

Container IPs are **ephemeral** — a restarted or rescheduled container usually
gets a new one. Hard-coding IPs would break constantly, so we use **DNS**:
stable *names* that resolve to current addresses. Inside a cluster, a Service
named `payments` is reachable simply as `payments` — Kubernetes runs an internal
DNS server that keeps the name pointed at the right place. This indirection is
what makes a fleet of disposable containers usable.

## Load balancing — one name, many backends

When several identical copies of an app run, a **load balancer** spreads
incoming connections across them. That gives you both *scale* (more copies =
more capacity) and *resilience* (one copy dies, traffic flows to the rest). A
Kubernetes Service is essentially a stable name plus a built-in load balancer
over a changing set of healthy Pods.
