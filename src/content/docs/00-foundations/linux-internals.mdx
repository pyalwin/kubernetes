Here is the secret that demystifies containers: **a container is just a normal
Linux process.** There is no tiny virtual machine inside. When you run a
container, the kernel starts an ordinary process and then *restricts what it can
see and use* with two long-standing Linux features. Understanding them turns
Kubernetes networking, security, and resource limits from magic into mechanics.

## Namespaces — what a process can *see*

A **namespace** gives a process its own private view of some system resource.
The kernel offers several kinds:

- **PID** namespace: the process sees its own process tree — inside, its app is
  PID 1 and it cannot see the host's other processes.
- **Network** namespace: its own network interfaces, IP, and routing table.
- **Mount** namespace: its own filesystem mounts — this is how a container gets
  its own root filesystem.
- **UTS / IPC / user** namespaces: own hostname, own shared-memory, own user-id
  mapping.

Stack these together and a process genuinely feels like it has the machine to
itself, while actually sharing the host kernel.

## cgroups — what a process can *use*

Namespaces isolate *visibility*; **cgroups** (control groups) limit
*consumption*. A cgroup caps and accounts CPU, memory, and I/O for a group of
processes. This is exactly what Kubernetes resource `requests` and `limits`
compile down to — a Pod's memory limit is, ultimately, a cgroup setting.

## The filesystem is layers

A container's root filesystem is the image's read-only layers stacked by a
**union filesystem**, with one thin writable layer on top. Writes go to that top
layer and vanish when the container is removed — which is why container storage
is *ephemeral* by default, and why Part 1's Volumes chapter exists.
