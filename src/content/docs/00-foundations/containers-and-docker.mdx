Kubernetes runs containers, so the container is where we start. A **container**
packages an application together with everything it needs to run — libraries,
binaries, config — into one isolated, portable unit. The promise is "it runs the
same on my laptop, in CI, and in production," because the environment travels
*with* the app instead of being assembled around it on each machine.

## Images vs. containers

These two words get mixed up constantly, so pin them down now:

- An **image** is the immutable template — a stack of read-only filesystem
  layers plus metadata (what command to run, which ports, which env vars).
- A **container** is a *running instance* of an image, with a thin writable
  layer on top. You can start many containers from one image, the way you start
  many processes from one program.

## Dockerfile → image → registry → running container

The usual lifecycle:

1. You write a **Dockerfile** — a recipe of steps (`FROM`, `COPY`, `RUN`,
   `CMD`) that describes how to build the image.
2. You **build** it into an image. Each instruction adds a cached layer, so
   rebuilds only redo what changed.
3. You **push** the image to a **registry** (Docker Hub, GHCR, ECR…) — a
   server that stores and serves images by name and tag.
4. A machine **pulls** the image and **runs** it on a **container runtime**.

## Docker vs. the runtime Kubernetes uses

Docker is the tool most people learn first, but Kubernetes does not actually
need Docker. It talks to a lower-level **container runtime** — most commonly
**containerd** — through a standard interface (the CRI). Docker itself uses
containerd under the hood. The mental model: Docker is a friendly toolbox for
building and running images; containerd is the lean engine that just runs them,
which is all a cluster node needs.
