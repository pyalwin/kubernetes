Production traffic is never flat. A storefront is quiet at 4am and on fire during
a sale; a batch job spikes then idles. Kubernetes answers this with **three
different autoscalers**, each acting on a different dimension. Confusing them is
the most common autoscaling mistake, so it is worth being precise.

## Horizontal Pod Autoscaler — more copies

The **HorizontalPodAutoscaler (HPA)** changes the *number of Pod replicas*. You
declare a target, most often average CPU utilization, and the HPA runs a control
loop: read the current metric, compare it to the target, and adjust the replica
count to close the gap. If you target `50%` CPU and your Pods are averaging
`90%`, it scales out; when they fall to `10%`, it scales back in.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 50 }
```

The HPA can also scale on memory or **custom/external metrics** (requests per
second, queue depth), which usually track user-facing load far better than CPU.

## Vertical Pod Autoscaler — right-sized Pods

The **VerticalPodAutoscaler (VPA)** does not add replicas; it tunes each Pod's
`requests` and `limits` so they match real usage. If you guessed `100m` CPU but
the app actually needs `400m`, the VPA recommends (and optionally applies) the
correction. This stops both *over-provisioning* (wasted, expensive headroom) and
*under-provisioning* (throttling and OOM kills). Because applying a new size
restarts the Pod, the VPA is usually paired with steady workloads — and you
generally do **not** run a CPU-based VPA and a CPU-based HPA on the same metric,
since they would fight.

## Cluster Autoscaler — more machines

Pods need somewhere to run. The **Cluster Autoscaler** watches for Pods stuck in
`Pending` because no node has room and **adds nodes** to the pool; when nodes sit
underused, it drains and **removes** them to save money.

## How they interact

The pieces form a chain. The HPA asks for more Pods; if those Pods can't be
scheduled, the Cluster Autoscaler grows the cluster; the VPA keeps each Pod's
requests honest so the other two reason about real numbers. All of this depends
on **metrics-server**, the lightweight component that collects per-Pod CPU and
memory and serves it through the metrics API — without it, `kubectl top` and the
HPA have nothing to read.
