Before we can talk about containers *finding* each other, we need a clear picture
of what it even means for two programs to talk over a network. This is foundational
plumbing — most of the time it's invisible — but the whole chapter rests on it, so
let's build it from the ground up in one section.

When one program wants to send data to another across a network, two numbers do
the addressing:

- An **IP address** identifies a *host* — a machine, or in our world, a
  container's network namespace. Something like `10.1.2.34`. It answers "*which*
  endpoint on the network?" Routers know how to shuttle packets toward an IP the
  way the postal system moves mail toward a street address.
- A **port** identifies a *program* on that host. A single container might run a
  web server on port 80 and a metrics endpoint on port 9090. The port answers
  "once the packet arrives, *which* listener gets it?" — the apartment number
  after the street address.

So the minimum a client needs is an **(IP, port)** pair: where the host is, and
which door to knock on. On top of that addressing sits **TCP**, the protocol that
turns a lossy, out-of-order packet network into something programs can actually
use: a reliable, ordered stream of bytes. TCP handles the messy reality —
retransmitting dropped packets, reordering ones that arrive scrambled, and slowing
down when the network is congested — so the application just reads and writes a
stream. (Its sibling **UDP** skips those guarantees for speed; DNS itself, which
we'll meet shortly, often rides on UDP.)

A live connection is pinned by *four* numbers — source IP, source port,
destination IP, destination port — which is how one server can hold thousands of
simultaneous conversations on the same port 80 without mixing them up: each
connection's quartet is unique.

Hold onto the destination half of that quartet — **destination IP and port** —
because that's precisely the part that's about to become a problem. In a cluster,
the *program* you want stays conceptually the same ("the web backend"), but the
*IP* it lives at refuses to hold still.

</Section>

Here's a clean idea that comes with a sting. In Kubernetes, **every Pod gets its
own IP address** — a real, routable one, drawn from the cluster's Pod network.
This is called the *IP-per-Pod* model, and it's deliberately simple: a Pod looks
like a little host on the network. No fighting over shared ports on a node, no
remembering which container got remapped to which host port. Each Pod just *has*
an address, and containers inside it share that address (recall from Chapter 3:
they share a network namespace).

Now collide that with the central lesson of Chapter 1: Pods are **cattle, not
pets.** They get killed and replaced constantly — a node drains, a deploy rolls,
a liveness check fails, the scheduler rebalances. And here's the catch: when a
controller replaces a dead Pod, the replacement is a *brand-new Pod*. New name,
new identity, and — critically — **a new IP address.** Nothing reuses the old
one. The self-healing you wanted is exactly what keeps the address moving.

So picture the failure. Your web Pod is told "the database lives at `10.1.2.34`,"
and it dutifully stores that. The database Pod crashes; a controller replaces it;
the new database Pod comes up at `10.1.5.91`. Your web Pod, none the wiser, keeps
firing packets at `10.1.2.34` — an address that now hosts *nothing*, or, after
enough churn, some *unrelated* Pod that just happens to have been handed that
recycled IP. Connection refused, or quietly talking to the wrong service. Either
way, broken.

Run the session above and watch it happen: `get pods -o wide` shows the Pod IPs;
delete one; list again and the address has changed. This is not a bug to be fixed
by making Pods stable — their disposability is the whole point of the model. The
fix has to live somewhere else: we need a layer of **indirection** between "the
thing I want to reach" and "the IP it currently happens to live at." A stable
name in front of a moving target.

</Section>

The fix is a principle older than Kubernetes, older than the cloud: *any problem
can be solved by adding a layer of indirection.* If the IP won't hold still, stop
handing out IPs. Hand out a **name** that never changes, and keep a lookup table —
maintained by something that's always watching — that maps the name to whatever
IPs are valid *right now*.

You already rely on this everywhere. You type `example.com`, not an IP address.
The website can move to entirely new servers with new addresses and you never
notice, because **DNS** — the Domain Name System — translates the stable name to
the current IP at the moment you connect. The name is the promise; the IP is an
implementation detail that's free to change behind it.

Kubernetes runs a DNS service *inside* the cluster (commonly CoreDNS) doing
exactly this job for your workloads. Instead of telling your web Pod "the database
is at `10.1.2.34`," you tell it "the database is at `db`." When the web Pod opens
a connection to `db`, it first asks cluster DNS, "what IP is `db`?" DNS answers
with a current, valid address, and *then* the TCP connection from the first
section is made to that address. The Pod code never holds an IP for longer than a
single connection; it holds a *name*.

<Callout type="note">
Cluster DNS gives every Service a predictable name. Within a namespace you can
reach one as just `db`; across namespaces the fully-qualified form is
`db.<namespace>.svc.cluster.local`. The shape never depends on where the Pods
landed or what addresses they drew — that's the stability you're buying.
</Callout>

But notice DNS alone leaves one question open. It resolves a name to an IP — fine
when there's exactly one target. In Chapter 1 we ran *three* replicas of the web
service for capacity and survival. So the name can't sensibly point at one Pod's
IP; that would funnel all traffic to a single replica and break the instant that
replica died. We need the name to front *a whole changing set* of Pods, and to
spread work across them. That's the final piece, and it's what turns a DNS entry
into a **Service**.

</Section>

The last piece upgrades a name-to-one-IP lookup into a name-to-*many* router. A
real service in production is a *set* of interchangeable replicas — three copies
of `web`, any of which can serve a request. The indirection layer therefore has
two jobs, not one: keep a **stable front** that clients address, and **spread
incoming connections** across whatever replicas are healthy at the moment, while
silently dropping ones that have died.

Kubernetes packages exactly this as a **Service**. A Service gives you:

- A **stable virtual IP** (the *ClusterIP*) and matching DNS name that *never*
  change for the life of the Service — the fixed thing clients hold onto. It does
  not belong to any Pod, so no Pod's death can take it down.
- A **live set of backend Pod IPs**, the Service's *endpoints*, kept current by a
  controller that watches Pods come and go. When a Pod with the matching label
  appears, its IP is added; when one dies, its IP is removed — automatically,
  using the very control-loop machinery from Chapter 1.
- **Load balancing**: a connection to the virtual IP is routed to *one* of the
  healthy endpoints. Traffic spreads across replicas, and a dead replica simply
  stops being a candidate.

Trace one request through Figure 4.2 and the whole chapter lines up. The client
holds a *name* (never changes). DNS resolves it to the Service's *virtual IP*
(never changes). The Service load-balances onto one of the *current Pod IPs*
(changes constantly, and that's fine — the client never sees it). Pods can crash,
scale from three to ten, or migrate to other nodes all day long; from the client's
seat, "talk to `web`" simply keeps working. The endpoint list churns underneath a
surface that stays perfectly still.

That stillness is the point. We started this chapter with a raw, moving Pod IP
and a client doomed to chase it. Three layers of indirection later — name, virtual
IP, load-balanced endpoint set — the moving target is completely hidden. The
client's world finally holds still, even though nothing underneath does.

</Section>
