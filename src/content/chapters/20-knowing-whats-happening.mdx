When everything works, you barely think about your cluster. The danger is the
other moment: it is 3am, the pager is screaming, error rates are climbing, and
the only question that matters is *what changed?* You cannot ship a debug build
and wait for a deploy. You have to ask questions of the system **as it is running
right now** — and get answers fast.

That ability has a name: **observability**. It is not a product you buy or a
single graph you stare at. It is a property of a system — the degree to which its
external outputs let you infer what is happening inside without stopping it or
modifying it. A system is observable when a question you didn't anticipate at
build time can still be answered at incident time.

You get there by emitting the right signals all along, while things are calm, so
they are waiting for you when things are not. Convention groups those signals into
**three pillars**, and the rest of this chapter is essentially those three plus
the machinery that moves them around.

</Section>

The three pillars are **logs**, **metrics**, and **traces**, and the reason there
are three is that each answers a question the others handle badly.

- **Logs** are timestamped, discrete events — `GET /api/orders 200 27ms`,
  `FATAL config missing`. They are the richest in detail: when you need to know
  *exactly* what happened to one request, a log line often has it verbatim. The
  cost is volume — searching billions of lines is slow and expensive.
- **Metrics** are numeric measurements sampled over time — CPU percent, requests
  per second, error ratio. They are cheap to store and aggregate, which makes
  them ideal for dashboards, trends, and the thresholds that trigger alerts. What
  they *don't* give you is the story of any single request.
- **Traces** follow one request as it travels across services, recording a timed
  **span** for each hop. When a metric tells you "p99 latency doubled" and a log
  tells you "this request was slow," a trace tells you *which service in the
  chain* spent the time.

<Callout type="note">
Think of it as zoom levels. Metrics are the wide shot — something is wrong, and
roughly where. Traces narrow to the request path and the slow hop. Logs are the
close-up — the exact error on the exact line. You move between them during an
incident, and a good setup makes that movement fast.
</Callout>

</Section>

<Section

  title="Metrics, the Prometheus way"
  brief={{

      { text: "It reads them straight out of etcd." },

The de-facto standard for metrics in Kubernetes is **Prometheus**, and its
defining choice is that it **pulls**. Each workload exposes a plain HTTP endpoint
— conventionally `/metrics` — that prints its current numbers in a simple text
format. Prometheus discovers targets from the Kubernetes API and **scrapes** each
endpoint on a fixed interval, storing every sample as a time series.

Pull sounds like a detail; it changes the failure model. Because Prometheus
initiates each scrape, a target that goes silent produces a *failed scrape*,
which is itself an observable event — you learn a Pod is unreachable for free.
Apps never need to know the monitoring system's address, and adding a new replica
needs no configuration: it appears via service discovery and gets scraped
automatically.

You ask questions of the stored series with **PromQL**:

```text
# requests per second over 5 minutes, by route
sum(rate(http_requests_total[5m])) by (route)

# error ratio: 5xx responses over all responses
sum(rate(http_requests_total{code=~"5.."}[5m]))
  / sum(rate(http_requests_total[5m]))
```

`rate(...[5m])` turns an ever-climbing counter into a per-second rate; `sum by`
collapses the labels you don't care about. **Alerting** is the same language with
a clock attached: a rule fires when an expression stays true for a duration — say
error ratio `> 5%` for ten minutes — and Alertmanager routes the resulting page
to the right people. Dashboards and alerts are just saved PromQL.

</Section>

<Section

  brief={{

  }}
  quiz={{
    question: "Why run the log collector as one agent per node rather than one per Pod?",
    options: [
      { text: "Pods aren't allowed to write logs." },
      { text: "A node agent tails the stdout of every container on its node, so collection scales with machines and survives Pods coming and going.", correct: true },
      { text: "Per-node agents are required by etcd." },
    ],
  }}
>

Kubernetes meets this model halfway, supplying just enough plumbing that the
tools above have something to read.

For **metrics**, the lightweight **metrics-server** (the same one the HPA depends
on) collects per-Pod CPU and memory and serves it through the metrics API. That
is the floor; your application metrics still come from your own `/metrics`
endpoint, scraped by Prometheus.

For **logs**, Kubernetes leans on a simple convention: containers write to
**stdout/stderr**, and the kubelet captures that stream to files on the node.
Collection then uses a **node-agent** pattern — one log shipper (Fluent Bit,
Vector, and friends) runs on every node, usually as a DaemonSet, tailing the
container log files for *all* Pods on that node and forwarding them to a central
store. The win is that collection scales with the number of **nodes**, not Pods,
and keeps working as Pods are created and destroyed underneath it.

Two commands let you read these signals directly. `kubectl top` queries
metrics-server for live resource use; `kubectl logs` streams a container's
stdout. Try both — `kubectl top pods`, then `kubectl logs` on one of them:
