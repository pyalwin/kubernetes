The fastest way to understand a container is to be clear about what it *isn't*.
When people first meet containers, they reach for the nearest familiar thing — a
virtual machine — and assume a container is just a smaller, faster one. That
guess is wrong in a way that matters, and unlearning it now saves you a dozen
confusions later.

A **virtual machine** is a pretend computer. A hypervisor carves out virtual
hardware — a fake CPU, fake disk, fake network card — and on top of that hardware
you boot an entire operating system, kernel and all. Two VMs on one server are
two complete, separate kernels that happen to share some silicon. That is why a
VM takes tens of seconds to boot and hundreds of megabytes of memory just to
exist: you are starting a whole computer.

A **container** boots nothing. There is no second kernel, no virtual hardware,
no operating system starting up. When you run a container, the host kernel — the
one already running on the machine — simply starts a normal process, the same
way it starts your text editor or your shell. That process shows up in the host's
process table with an ordinary PID. You can see it. You can `kill` it.

So why does the process *feel* like it lives in its own little world — its own
files, its own network, its own list of running programs? Because the kernel has
agreed to lie to it. Through a handful of features we'll name shortly, the kernel
hands that one process a private, curated view of the system and quietly enforces
limits on what it can touch. The isolation is real, but it is *isolation within a
shared kernel*, not a second computer.

<Callout type="warning" title="Why the distinction earns its keep">
Because containers share the host kernel, they start in milliseconds and cost
almost nothing to keep idle — that is the whole reason you can pack dozens onto a
node. But it also means a kernel-level exploit can, in principle, reach across
containers in a way it cannot reach across VMs. "Process, not machine" explains
both the speed *and* the security caveat in one stroke.
</Callout>

</Section>

There is a vocabulary trap that trips up nearly everyone, and clearing it up is
worth a whole section. The word "container" gets used for two genuinely
different things: the *thing on disk you download* and the *thing that runs*.
Keeping them separate makes the rest of this book easier.

An **image** is a template. It is an immutable, read-only package sitting on
disk: a frozen snapshot of a filesystem — application binary, libraries, config
files — plus a little metadata saying which command to run and which environment
to set. An image does nothing. It has no PID, no memory, no CPU usage. It is
inert, like a `.zip` you haven't opened, or a recipe card that has never met a
kitchen.

A **container** is what you get when the kernel brings an image to life: a
running (or paused, or stopped) *instance* created from that template. The kernel
takes the image's frozen filesystem, gives it the disguises from the previous
section, and starts the command the image specified. *That* is the process with a
PID.

The relationship is exactly the one programmers know as class-to-object: the
image is the class, each container an instance. Run one image five times and you
get five containers — five independent processes, each with its own memory and
its own scratch space — all reading from the *same* unchanging template. One
recipe, five meals.

This is why **immutability** is such a load-bearing idea. Because the image never
changes, a container started from `nginx:1.27` on your laptop is byte-for-byte
the same template as the one a colleague starts in a datacenter. "Works on my
machine" stops being a shrug and becomes a guarantee. And when Kubernetes needs
to replace a crashed copy, it doesn't repair anything — it throws the instance
away and stamps out a fresh one from the same template. Cattle, not pets. Hold
onto that: the control loop from Chapter 1 can only "just make another one"
because the template it stamps from never drifts.

<Section
  id="layers"
  title="How images are built: layers"
  brief={{
    problem: "If every image were one monolithic blob, downloading and storing hundreds of nearly-identical images would waste enormous bandwidth and disk.",

  }}
  quiz={{
    question: "Ten images all built 'FROM ubuntu:24.04'. How much disk does that shared base consume?",
    options: [
      { text: "Ten copies — one full base per image." },
      { text: "One copy — the shared base layer is stored once and referenced by all ten.", correct: true },
      { text: "None — base layers are recomputed on every run." },
    ],

>

An image isn't a single sealed blob. It is a **stack of layers**, and that design
choice is the reason container workflows feel light instead of crushing.

Each layer is a set of filesystem changes — a diff. The bottom layer might be a
minimal Linux userland (`FROM ubuntu:24.04`). The next layer is whatever changed
when you installed your language runtime. The next is your application's
dependencies. The top is your own code. Stack them in order, merge them, and you
get a complete root filesystem — but it is *built up* from independently stored,
read-only slices.

Two properties fall out of this, and both are why the ecosystem works at scale:

- **Layers are shared.** That `ubuntu:24.04` base is content-addressed: identified
  by a hash of its bytes. Pull a hundred images built on it and the base is
  stored on disk *once* and transferred over the network *once*. Every image that
  references that hash reuses the same copy. Disk and bandwidth stay flat instead
  of multiplying.
- **Only changes move.** Rebuild your app after a one-line code edit and only the
  top layer changes; its hash is new, every layer below it is unchanged. Pushing
  the new image uploads a few kilobytes, not the whole hundreds of megabytes. The
  registry already has the rest.

A natural question: if all the layers are *read-only*, how does a running
container write a log file or a temp file? When the kernel starts a container, it
adds one more layer on top — a thin, **writable** layer that belongs to that
single instance. Reads fall through the stack to wherever the file lives; writes
land in the top layer, leaving the shared read-only layers pristine. That is also
why deleting a container is cheap and safe: you discard its tiny writable layer,
and the shared template underneath is untouched, ready for the next instance.

We'll see in the next chapter that this stacking is performed by a real kernel
feature — a *union filesystem* — not by copying files around. For now the picture
is enough: an image is layers; layers are shared diffs; a running container adds a
private writable layer on top.

</Section>

>

Two more pieces and the container supply chain is complete: how an image is
*built*, and where it *lives*.

A **Dockerfile** is the build recipe — a short, ordered script. Each significant
instruction produces exactly one layer, which ties directly back to the previous
section:

```dockerfile
FROM ubuntu:24.04            # layer 0: the base userland
RUN apt-get install -y nginx # layer 1: install the web server
COPY ./site /var/www/html    # layer 2: add your content
CMD ["nginx", "-g", "daemon off;"]  # metadata: what to run
```

You run a build tool over this, and it executes each step in a throwaway
container, snapshots the filesystem diff as a layer, and moves on. The ordering
matters for caching: put the things that rarely change (the base, the
dependencies) near the top and the things that change constantly (your code) near
the bottom, and most rebuilds reuse every cached layer but the last.

The result is an image, named like `myregistry.io/web:1.4` — a **name**, a
**tag** (`1.4`), and underneath, a content hash that pins the exact bytes. But an
image on your laptop helps no one else. To share it, you push it to a
**registry**: a server whose entire job is to store images and serve them by
name. Docker Hub is the public default; cloud providers and companies run private
ones. `docker push` uploads your layers (skipping any the registry already has);
`docker pull` downloads them by name.

This is the supply chain Kubernetes lives downstream of. When you tell the
cluster to run `web:1.4`, it does not build anything — somewhere a node's runtime
issues a pull against the registry, fetches the layers it's missing, assembles
the root filesystem, and starts the process. The registry is the warehouse; the
node is the floor where the template becomes a running thing.

<Callout type="tip">
Prefer a specific tag (`web:1.4`) or, better, a content digest over the floating
`latest` tag. `latest` means "whatever was pushed most recently," which can
change under you between two `pull`s on two nodes — quietly breaking the
byte-for-byte sameness that made images trustworthy in the first place.
</Callout>

</Section>

We've assembled the parts — process, image, layers, registry — but something has
to do the actual work of turning a name into a running process. That something is
a **container runtime**, and seeing where it sits explains a piece of Kubernetes
trivia that confused a lot of people in 2020.

When a node is told to run `web:1.4`, the runtime is what acts. It pulls the
image's layers from the registry, stacks them into a root filesystem, asks the
kernel to set up the isolation (the disguises from this chapter, detailed in the
next), and starts the process. On most clusters today that runtime is
**containerd**. Below it sits an even lower-level piece, `runc`, that makes the
final kernel calls to bring the container into existence. It's runtimes all the
way down, each layer doing a smaller, more concrete job.

Now the design question Kubernetes had to answer: it must not be welded to one
runtime. Tying the scheduler to a single vendor's tool would be a strategic dead
end. So Kubernetes defines a contract — the **Container Runtime Interface
(CRI)** — a standard API that says "here is how the cluster will ask you to pull
an image, create a container, start it, stop it." *Any* runtime that implements
the CRI can plug in: containerd, CRI-O, others. The cluster doesn't know or care
which; it only knows the contract.

This seam is the whole reason "Kubernetes is deprecating Docker" caused a panic
and then turned out to be a non-event. Kubernetes had never spoken to Docker
directly for very long; it spoke through the CRI, and Docker's daemon needed a
little adapter (`dockershim`) to fit that contract. Removing the adapter changed
nothing about your images or your Pods — `containerd`, which Docker itself uses
under the hood, kept doing the work. The contract held; the implementation behind
it swapped out. That is exactly the kind of clean seam the rest of Kubernetes is
built from.
