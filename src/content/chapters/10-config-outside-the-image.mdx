You have one container image for your app. It needs to run in three places — your
laptop, a staging cluster, and production — and in each place it needs *different*
settings: a different database URL, a different log level, different API
credentials. Where do those settings go?

The tempting answer is "inside the image." Drop a `config.prod.json` into the
build, or hard-code the database host. It works on day one and rots immediately.
An **image is a build artifact** — an immutable, content-addressed stack of
layers you build once and promote. **Configuration is environment-specific** — it
changes per place, sometimes per deploy. The instant you mix the two, you've
broken a core promise: that the artifact you tested in staging is byte-for-byte
the one running in prod.

Concretely, baking config in costs you:

- **A rebuild per environment.** Want to change the prod database host? Rebuild
  and re-push the image. Now "staging" and "prod" are *different images*, and the
  whole point of a tested artifact evaporates.
- **Leaked secrets.** Put a password in the image and it's in a layer. Anyone who
  can pull the image — your whole CI system, every developer, the registry — can
  unpack that layer and read it. Image layers are not a vault.

The fix is to keep the image *pure* — just code and dependencies — and inject the
environment-specific bits at runtime. Kubernetes gives you two objects for exactly
this, split by one question: is the data sensitive?

</Section>

The first object is the **ConfigMap**, and it holds the *non-sensitive* half of
your configuration: log levels, feature flags, upstream hostnames, timeouts, even
entire config files. It's just a named bundle of key-value data living in the
cluster, created and edited independently of any image.

You make one with a manifest or a command like `kubectl create configmap
app-config --from-literal=LOG_LEVEL=info`, and from then on it's a first-class
object your Pods can reference. Crucially, the *image* knows nothing about it. The
same `web:v1` image runs in staging with a staging ConfigMap and in prod with a
prod ConfigMap — one artifact, two configurations, zero rebuilds. Inspect one:

Notice what this buys you organizationally. Configuration becomes a thing you can
review, diff, and roll back on its own, separate from code. A platform team can
own the prod ConfigMap while developers own the image. And because it's a normal
Kubernetes object, it goes through the same API server, the same RBAC, the same
audit trail as everything else. The settings stopped being a property of the
build and became a property of the *deployment* — which is exactly where
environment-specific things belong.

</Section>

The second object is the **Secret**, for the *sensitive* half: passwords, API
tokens, TLS private keys. Structurally it's almost identical to a ConfigMap — a
named bundle of key-value data — but Kubernetes treats it with more care, and you
must understand exactly *how much* more, because the most common misconception in
this whole area lives here.

**A Secret is base64-encoded, not encrypted.** Look back at the terminal in the
previous section: `username: YWRtaW4=` is just `base64("admin")`. Anyone who can
read the Secret object can decode it in one command. Base64 is there so binary
data (like a certificate) survives being stored as text — it is *encoding*, a
reversible transformation with no key, not *security*. Treating a Secret as
inherently safe because "it's encoded" is how credentials leak.

<Callout type="warning">
By default, Secret data sits in etcd essentially in the clear. To get real
protection you must (1) enable **encryption at rest** so etcd encrypts Secret
values on disk, and (2) lock down **RBAC** so only the workloads and people that
need a Secret can read it. Many teams also pull secrets from an external manager
(Vault, cloud KMS) rather than storing long-lived ones in the cluster at all.
</Callout>

What Kubernetes *does* do for Secrets beyond ConfigMaps: it can keep them in
memory (tmpfs) rather than on a node's disk when mounted, it avoids printing their
values in logs and `describe` output, and it lets you scope access tightly. Use a
Secret for anything sensitive — but know that the object is a *container* for
secret data, and the actual confidentiality comes from the controls you put around
it.

</Section>

<Section
  id="env-vs-files"
  title="Two ways in: environment variables vs. files"
  brief={{
    problem: "A ConfigMap or Secret in the cluster is useless until its values reach the process inside the container. How they're injected changes how your app reads them and whether updates take effect live.",
    mechanism: "Inject either as environment variables (read once at startup) or as mounted files in a volume (each key a file, can update live).",
    takeaway: "Env vars are simple but frozen at start; mounted files suit config files and can refresh without a restart.",
  }}
  quiz={{
    question: "You mount a ConfigMap as files and later edit the ConfigMap. What's the advantage over injecting it as env vars?",
    options: [
      { text: "Files load faster than environment variables." },
      { text: "Mounted files can update in place without restarting the Pod, whereas env vars are fixed at container startup.", correct: true },
      { text: "Only files can hold non-sensitive data." },
    ],
  }}
>

Having a ConfigMap or Secret in the cluster does nothing until its values reach
the process inside the container. There are two ways to inject them, and the
choice has real consequences.

**As environment variables.** You map each key to an `env` entry, and the process
reads it at startup like any environment variable — `process.env.LOG_LEVEL`,
`os.environ["DB_PASSWORD"]`. Simple and universal. The catch: environment
variables are read *once*, when the container starts. Change the underlying
ConfigMap and the running process won't see it until it restarts. Env vars are
also easier to leak — they show up in child processes, crash dumps, and careless
logging — which is why they're a weaker choice for secrets.

**As mounted files.** You mount the ConfigMap or Secret as a volume, and each key
becomes a file in a directory — `/etc/config/LOG_LEVEL` containing `info`. This
fits apps that already read a config file, and it has a quiet superpower: when you
update the ConfigMap, the mounted files update in place (after a short sync
delay), so a process that re-reads its config can pick up changes *without a
restart*. For Secrets, file mounts are generally preferred over env vars — files
are less likely to leak into logs and child processes.

A rough rule: reach for **env vars** for a handful of simple scalar settings, and
for **file mounts** when you have a real config file, when values might change
under a running Pod, or when the data is a secret you want to keep out of the
environment. Either way, the win is the same one from the top of the chapter — the
image stays pure, and the configuration arrives at runtime.

That covers a Pod's *settings*. But settings are small and disposable. What about
real **data** — the rows a database must not lose when its Pod is rescheduled to
another machine? That's a harder problem, and it's next.
