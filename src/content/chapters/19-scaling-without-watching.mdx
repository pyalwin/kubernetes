Picture a storefront. At 4am it serves a trickle; two copies of the app are
plenty. At 9am a promotion goes live and traffic is ten times higher. If you
sized the Deployment for the quiet hours, the morning melts it down. If you sized
it for the spike, you pay for eight idle copies all night.

The obvious fix is the wrong one: *watch a graph and edit `replicas` when it gets
busy.* You already know how this ends — it is the same hand-cranked control loop
that motivated the whole system. Someone has to be looking. Someone has to react
in seconds. Someone has to be awake.

So we want the cluster to do what it already does for *failures* — observe,
compare, act — but pointed at **load** instead. That turns out to need not one
autoscaler but three, because "scale" means three different things: more copies
of the app, a better-fitting size for each copy, and more machines to put them
on. The rest of this chapter takes them one at a time, then shows how they snap
together.

</Section>

>

The **HorizontalPodAutoscaler** — HPA — is the one most people mean when they say
"autoscaling." It changes the *number* of Pod replicas. You give it a target,
most often average CPU utilization, and it runs the familiar loop: read the
current metric, compare it to the target, and adjust the replica count to close
the gap.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef: { kind: Deployment, name: web }
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 50 }
```

Read it as a sentence: *keep `web` between 2 and 20 copies, sized so the average
Pod sits near 50% CPU.* When the morning spike pushes Pods to 90%, the HPA adds
replicas until the average falls back toward target; when the rush passes, it
removes them. CPU is only the default — the HPA can also scale on memory or, far
more usefully, on **custom and external metrics** like requests per second or
queue depth, which track user-facing load much more honestly than CPU does.

<Callout type="tip">
The HPA does not scale on a whim. It uses a stabilization window and a
ratio-based formula so a brief blip doesn't thrash the replica count up and down.
Smooth is the goal: follow the trend, ignore the noise.
</Callout>

</Section>

      { text: "The number of replicas in the Deployment." },
      { text: "The CPU and memory requests/limits of each Pod, to match its actual usage.", correct: true },
      { text: "The number of nodes in the cluster." },
    ],

>

Horizontal scaling assumes each Pod is the *right size*. Often it isn't. You
guessed `100m` of CPU at design time; the app actually wants `400m`. Now it
throttles under load and the HPA, reading a misleadingly high utilization, scales
out to paper over a sizing mistake. Guess too *high* instead and every replica
reserves capacity it never uses, so the scheduler packs fewer Pods per node and
you burn money.

The **VerticalPodAutoscaler** — VPA — fixes the size rather than the count. It
watches a workload's real consumption over time and recommends (or, in auto mode,
applies) corrected `requests` and `limits`. Because changing a running Pod's
resources means recreating it, the VPA is best on steady, long-lived workloads
where the occasional restart is cheap.

<Callout type="warning">
Do **not** point a VPA and an HPA at the *same* resource metric. If both react to
CPU, they fight — the VPA grows the Pod, which changes utilization, which
confuses the HPA. Pair them only when they watch different signals (e.g. VPA on
memory, HPA on requests-per-second), or pick one.
</Callout>

</Section>

    takeaway: "The Cluster Autoscaler changes how many machines exist, closing the loop the HPA opens.",
  }}
  quiz={{
    question: "The HPA scaled to 20 replicas but several Pods are stuck Pending. Who resolves it?",
    options: [
      { text: "The HPA forces the extra Pods onto already-full nodes." },
      { text: "The Cluster Autoscaler sees the unschedulable Pods and adds a node so they can be placed.", correct: true },
      { text: "metrics-server reschedules them automatically." },
    ],
  }}
>

There is a ceiling the HPA can't break through on its own. It can *request* 20
Pods, but if the existing nodes are full, the scheduler has nowhere to put the
extras and they sit in `Pending`. More copies you can't place are not more
capacity.

The **Cluster Autoscaler** operates one level down, on the machines themselves.
Its loop watches for Pods that can't be scheduled and, when it finds them, asks
the cloud provider for another node and joins it to the cluster so the Pods land.
Running the loop in reverse, when a node has been underused long enough and its
Pods can fit elsewhere, it cordons the node, drains those Pods, and deletes the
machine — so the bill tracks real demand instead of peak demand.

</Section>

Step back and the three line up into a single pipeline. The **VPA** keeps each
Pod's requests honest, so the numbers everyone else reasons about are real. The
**HPA** decides *how many* of those right-sized Pods you need for the current
load. And when that number of Pods won't fit, the **Cluster Autoscaler** adds the
machines to hold them. Each loop closes a gap the loop above it opens.

Underneath all of it sits one humble dependency: **metrics-server**. It is a
lightweight component that scrapes per-Pod CPU and memory from each node's
kubelet and serves it through the metrics API. It is what `kubectl top` reads,
and it is what the HPA reads to make every decision. No metrics-server, no
horizontal autoscaling and no `top` — the loops would be blind. Install it once,
and the whole stack above can finally watch load so you don't have to.

</Section>
