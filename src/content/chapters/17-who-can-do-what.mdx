You learned early that the API server is the cluster's single front door —
*everyone*, human or controller, goes through it. That design is wonderful for
observability and terrible if there's no gate, because it means a single door
stands between any authenticated client and *everything*: every Secret, every
Deployment, every node.

So Kubernetes puts a guard at that door. After it has **authenticated** a request —
worked out *who* is asking — it must **authorize** it: decide whether that identity
may perform *this* action on *this* object. The dominant mechanism for that decision
is **RBAC**, Role-Based Access Control, and the whole point of getting it right is
**containment**. When a workload is compromised — and eventually one will be — RBAC
is what decides whether the attacker can read one namespace's ConfigMaps or walk off
with every Secret in the cluster.

The mental model is a sentence the API server completes for every request: *Can this
**subject** perform this **verb** on this **resource**?* The rest of the chapter is
the vocabulary for filling in those three blanks, and the discipline — **least
privilege** — for filling them in as narrowly as possible.

</Section>

<Section

  brief={{

    options: [

>

RBAC's vocabulary is small and precise. A permission is a pairing of:

- **Verbs** — the actions: `get`, `list`, `watch`, `create`, `update`, `patch`,
  `delete`.
- **Resources** — the object kinds: `pods`, `secrets`, `deployments`, `services`,
  `nodes`.

A **Role** bundles these into a list of rules. Here's one that can do exactly two
things — read pods — and nothing else:

```yaml
kind: Role
metadata: { namespace: team-a, name: pod-reader }
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
```

A Role by itself grants nobody anything; it's just a named capability sitting on a
shelf. A **RoleBinding** takes it off the shelf and hands it to **subjects** — users,
groups, or (as we'll see) ServiceAccounts:

```yaml
kind: RoleBinding
metadata: { namespace: team-a, name: read-pods }
subjects:
  - kind: User
    name: alice
roleRef:
  kind: Role
  name: pod-reader
```

One property shapes everything about how you use RBAC: **it is purely additive.**
There are no "deny" rules. A subject can do exactly what some binding grants it, and
nothing else is permitted by default. So least privilege isn't about carving
exceptions out of a broad grant — it's about granting *little* in the first place.
You start from "can do nothing" and add only the specific verbs on the specific
resources a subject genuinely needs.

</Section>

<Section

  title="Namespace or cluster: Role vs ClusterRole"
  brief={{

      { text: "A Role is scoped to one namespace; a ClusterRole grants permissions cluster-wide and over cluster-scoped resources.", correct: true },
      { text: "There is no difference; they are aliases." },
    ],

>

Permissions come in two scopes, and RBAC gives you a matched pair for each.

A **Role** and its **RoleBinding** are **namespaced**. They grant permissions inside
a single namespace and nowhere else. "Read pods in `team-a`" is a Role in `team-a`;
it says nothing about pods in `team-b`. This is the right tool most of the time,
because most workloads and teams live within one namespace.

A **ClusterRole** and its **ClusterRoleBinding** apply **cluster-wide**. You need
them in two situations:

1. **Cluster-scoped resources.** Some objects don't live in a namespace at all —
   `nodes`, `namespaces`, `persistentvolumes`. Permissions over them *must* be a
   ClusterRole.
2. **The same permission across every namespace.** "Read pods in *all* namespaces"
   for a monitoring tool is a ClusterRole bound cluster-wide.

There's a useful hybrid worth knowing: you can reference a **ClusterRole** from a
namespaced **RoleBinding**. That lets you define a common capability once — say a
generic "view" ClusterRole — and grant it inside one namespace only. The ClusterRole
supplies the *what*; the namespaced RoleBinding supplies the *where*.

<Callout type="tip">
Default to the narrow tool. If a namespaced Role does the job, use it. Reaching for a
ClusterRoleBinding "to be safe" is exactly backwards — it's the broadest grant in the
system, and every one you create is a bigger thing to get wrong.
</Callout>

</Section>

<Section

  title="ServiceAccounts: identity for pods"
  brief={{

    takeaway: "Give each app its own ServiceAccount bound to only what it needs; never run workloads as cluster-admin.",
  }}
  quiz={{
    question: "What identity does a pod use when it calls the Kubernetes API?",
    options: [
      { text: "The node's root user." },
      { text: "A ServiceAccount, whose token is mounted into the pod.", correct: true },
      { text: "The cluster administrator's certificate." },
    ],
  }}
>

RBAC's subjects so far have been *users*, which are humans authenticating with
certificates or an identity provider. But pods need to talk to the API server too — a
controller reconciling objects, an app reading its own ConfigMaps, an operator
managing resources. A pod isn't a human and has no certificate. Its identity is a
**ServiceAccount**.

A ServiceAccount is a namespaced identity meant for in-cluster workloads. When a pod
runs as one, Kubernetes mounts a token for that ServiceAccount into the container,
and the pod presents it on every API call. The API server authenticates the pod as
that ServiceAccount — and then RBAC takes over, checking what that ServiceAccount is
allowed to do.

That's where least privilege becomes concrete. The recipe is always the same:

```yaml
kind: RoleBinding
metadata: { name: log-reader-binding, namespace: team-a }
subjects:
  - kind: ServiceAccount
    name: log-reader
roleRef:
  kind: Role
  name: pod-reader   # only get/list/watch pods
```

Create a **dedicated** ServiceAccount per app, bind it to a Role granting *only* what
that app needs, and run the pod as it. Now if the app is compromised, the attacker
inherits only those narrow permissions.

<Callout type="warning">
Every pod that doesn't specify a ServiceAccount gets the namespace's `default` one —
and people are tempted to bind broad powers to `default` "to make things work." Don't.
And never bind a workload to the built-in `cluster-admin` ClusterRole: a single
breached pod with cluster-admin is a breached *cluster*.
</Callout>

</Section>

<Section
  id="pod-security"
  title="Pod security: hardening the container itself"
  brief={{
    problem: "RBAC controls what a pod can do through the API, but a compromised process can still try to abuse the node it runs on — root access, privilege escalation, dangerous capabilities.",
    mechanism: "A securityContext constrains the container at the kernel level; Pod Security Standards bundle these into baseline and restricted profiles enforced per namespace.",
    takeaway: "Defense in depth: RBAC guards the API, securityContext guards the kernel — use both.",
  }}
  quiz={{
    question: "What does setting runAsNonRoot and dropping all capabilities in a pod's securityContext achieve?",
    options: [
      { text: "It speeds up the container's startup." },
      { text: "It hardens the container at the kernel level, shrinking what a compromised process can do to the node.", correct: true },
      { text: "It grants the pod more API permissions." },
    ],
  }}
>

RBAC answers "what can this pod do to the *cluster*?" There's a second, lower
question it doesn't touch: "what can this pod do to the *node* it's running on?" A
container is, underneath, just a Linux process. If it runs as root, can escalate
privileges, or holds powerful kernel capabilities, a compromise inside it can become
a compromise of the host.

The fix lives on the pod itself: the **securityContext**. It constrains the container
at the kernel level:

```yaml
securityContext:
  runAsNonRoot: true            # refuse to run as root
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]               # surrender all Linux capabilities
  readOnlyRootFilesystem: true
```

Each line shrinks the blast radius. Not root, so it can't touch root-owned files on
the host. No privilege escalation, so it can't `setuid` its way up. No capabilities,
so it can't do the privileged kernel operations that container escapes rely on.

Setting this per pod by hand is easy to forget, so Kubernetes packages sensible
bundles as **Pod Security Standards**: `privileged` (unrestricted), `baseline` (blocks
the known-dangerous things), and **`restricted`** (the strict profile — non-root, no
privilege escalation, a tight capability set, and more). You apply a standard to a
*namespace* with a label, and the cluster enforces it on every pod admitted there:

```yaml
# on the namespace
labels:
  pod-security.kubernetes.io/enforce: restricted
```

Now a pod that asks to run as root in that namespace is simply rejected.

This is **defense in depth**, and it's the right note to end on. RBAC contains a
workload at the API; securityContext and Pod Security Standards contain it at the
kernel. Neither alone is enough — a pod with no API permissions can still attack its
node, and a hardened container with cluster-admin can still loot the cluster. Least
privilege at *both* layers is how you make a single compromise stay small.

</Section>
