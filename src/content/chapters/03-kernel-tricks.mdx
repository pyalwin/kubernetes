When we say a container is "isolated," we are quietly bundling two completely
different ideas, and prying them apart is the key that unlocks this whole chapter.

Ask yourself what isolation actually requires. First: the container shouldn't be
able to *see* the host's other processes, snoop on its network, or read its
files. That's a question about **visibility** — what exists in this process's
world. Second: the container shouldn't be able to *starve* its neighbors by
eating all the CPU or all the memory. That's a question about **consumption** —
how much of the machine it may use.

These are independent. A process could be perfectly blind to everything around it
and *still* consume every CPU cycle on the box — total visual isolation, zero
consumption limits. Or it could be capped to 5% CPU while still able to see and
signal every other process on the host — totally constrained consumption, zero
visual isolation. Neither alone is "a container." You need both, and because they
answer different questions, the kernel provides them as two different features:

- **Namespaces** answer *what can I see?* They give a process a private view of
  some slice of the system.
- **cgroups** (control groups) answer *how much can I use?* They meter and cap a
  process's resource consumption.

Hold this split in your head for the rest of the chapter; we'll take each one
apart in turn. The reason it pays off is that Kubernetes exposes both, separately,
in every Pod you ever write — and once you see that a `securityContext` is really
configuring namespaces and a `resources` block is really configuring cgroups, the
Pod spec stops being arbitrary YAML and becomes a thin remote control over these
two kernel levers.

  title="Namespaces: controlling what a process sees"
  brief={{

A **namespace** wraps one global system resource so that the processes inside the
namespace see their own isolated instance of it, while the host and other
namespaces are invisible. The kernel ships several types, and a container is
simply a process placed into a fresh bundle of them at once. The important ones:

- **PID namespace** — its own process-number space. The container's main process
  is PID 1 *inside*, even though the host kernel knows it as PID 4821. It cannot
  see, signal, or even prove the existence of host processes; its `ps` lists only
  its own.
- **Network namespace** — its own network stack: its own interfaces, its own IP
  address, its own routing table and ports. This is why two containers can each
  bind port 80 with no clash — they're port 80 on two *different* private stacks.
  (This is the foundation for everything in the next chapter.)
- **Mount namespace** — its own filesystem tree. The container's `/` is the
  layered image root from Chapter 2, not the host's `/`. It cannot wander into the
  host's files because, from inside, they aren't mounted.
- **UTS namespace** — its own hostname and domain name, so a container can call
  itself `web-1` without renaming the host.
- **IPC namespace** — its own inter-process-communication objects (shared memory,
  semaphores), so containers can't reach each other's IPC.
- **User namespace** — its own mapping of user and group IDs. This lets a process
  be `root` (UID 0) *inside* the container while mapping to an unprivileged,
  harmless UID *outside* — a major hardening win.

None of this is copying or emulation. There is one kernel, one real process
table, one real network stack. A namespace is a *filtered lens*: the kernel
tags the process and, whenever it asks "what processes exist?" or "what's my
IP?", answers using only what's tagged to its namespace. The same process, asked
the same question from inside versus outside, gets two honest but different
answers.

<Callout type="note">
Here's the Kubernetes payoff. A **Pod** is defined as one or more containers that
*share a subset of namespaces* — specifically the network and IPC namespaces.
That's the entire technical reason containers in a Pod reach each other on
`localhost` and share an IP: they're living in the *same* network namespace. The
Pod isn't a metaphor; it's a namespace-sharing boundary.
</Callout>

</Section>

Namespaces hide the neighbors; they do nothing to stop you from eating their
lunch. That job belongs to **control groups**, universally shortened to
**cgroups** — the kernel feature that meters and limits what a set of processes
may consume.

A cgroup is a labelled group of processes with resource controllers attached.
The kernel arranges them in a tree and, for each group, can both *account* ("how
much CPU has this group used?") and *enforce* ("this group may not exceed X").
The controllers that matter most:

- **CPU.** You can grant a group a *share* (relative weight under contention) and
  a hard *quota* (e.g. "at most 1.5 cores' worth of time per scheduling period").
  Exceed the quota and the kernel **throttles** the group — it's paused until the
  next period. Your process isn't killed; it's just made to wait, which shows up
  as mysterious latency if you're not expecting it.
- **Memory.** You set a ceiling in bytes. Allocate under it and all is well. Try
  to cross it and there's no negotiating: the kernel's **OOM (out-of-memory)
  killer** terminates a process in the group to bring usage back under the cap.
  This is the true story behind a container that keeps dying with status
  `OOMKilled`.
- **I/O.** You can cap block-device read/write bandwidth so one container can't
  saturate the disk and starve the rest.

The asymmetry between CPU and memory is worth internalizing because it explains a
lot of production behavior: **CPU is compressible, memory is not.** Over your CPU
budget? You wait — throttled, slow, but alive. Over your memory budget? You die —
there's no "wait" for memory the kernel has already promised elsewhere.

This is the machinery directly under Kubernetes' `resources` block:

```yaml
resources:
  requests:   # what the scheduler reserves for you when placing the Pod
    cpu: "250m"
    memory: "256Mi"
  limits:     # the cgroup caps actually enforced at runtime
    cpu: "1"
    memory: "512Mi"
```

`requests` are how the scheduler *bin-packs* — it promises each Pod that much
headroom and won't overcommit a node past what it can honor. `limits` become the
literal cgroup caps. When you read in Chapter 1 that a controller "just makes
another copy," cgroups are part of why that's safe: a misbehaving copy is boxed
in, throttled or OOM-killed, instead of dragging the whole node down with it.

</Section>

>

The third trick is the one that makes Chapter 2's layered images practical rather
than merely elegant. We said an image is a stack of read-only layers and a
running container adds a writable layer on top. The kernel feature that performs
that stacking — without copying the layers for every container — is a **union
filesystem**, most commonly **OverlayFS** on Linux.

A union filesystem takes several directories and presents them as one merged
tree. In overlay terms there are *lower* directories (the read-only image layers,
bottom to top) and an *upper* directory (the container's private writable layer).
When the container looks up a file, the kernel searches from the upper layer
downward and returns the first match — so an upper-layer file *shadows* a
same-named file below it. The container sees one ordinary, complete filesystem; it
has no idea it's a collage.

The clever part is what happens on a **write**, governed by **copy-on-write
(CoW)**:

- **Read a file** that exists only in a lower (read-only) layer: served straight
  from that shared layer. No copying.
- **Modify a file** that lives in a lower layer: the kernel first *copies it up*
  into the writable upper layer, then applies your change there. The original in
  the shared layer is never touched. Every other container reading that file still
  sees the pristine version.
- **Delete a file** from a lower layer: it can't actually be removed from a
  read-only layer, so the kernel writes a tiny "whiteout" marker in the upper
  layer that hides it from view.

Two things this buys you, both already promised in Chapter 2 and now paid for.
First, **sharing**: a hundred containers off the same image share its
read-only layers on disk exactly once; only their small upper layers differ.
Second, **disposability**: stopping a container and discarding its upper layer is
instant and harmless — the shared template underneath is pristine, ready for the
next instance. That's the mechanism that makes "cattle, not pets" cheap.

<Callout type="tip">
This is also why writing lots of data inside a container's own filesystem is a
trap: it piles up in the ephemeral upper layer and vanishes when the container
does. Persistent data belongs on a **volume** mounted in — a separate filesystem
that lives outside the layered stack and survives the container. The union
filesystem is for the *image*, not for your database.
</Callout>

</Section>

Step back and the payoff snaps into focus. Kubernetes did not invent a new way to
isolate workloads. It *orchestrates the three kernel tricks you just met*, and the
Pod spec is mostly a friendly dialect for asking the kernel to apply them. Read a
spec with that lens and it stops being a grab-bag of fields:

```yaml
spec:
  containers:
    - name: web
      image: web:1.4          # → union filesystem: which layers to stack
      resources:
        limits:
          cpu: "1"            # → cgroup CPU quota
          memory: "512Mi"     # → cgroup memory cap (cross it → OOMKilled)
      securityContext:
        runAsUser: 1000       # → user namespace mapping (not root)
        readOnlyRootFilesystem: true   # → mount namespace + overlay, read-only
  securityContext:
    fsGroup: 2000
```

Trace each line to its mechanism. The `image` selects the read-only layers the
**union filesystem** will stack. `resources.limits` become **cgroup** caps —
overshoot CPU and you're throttled, overshoot memory and the OOM killer ends you.
`securityContext.runAsUser` maps into a **user namespace**;
`readOnlyRootFilesystem` leans on the **mount namespace** and overlay to refuse
writes to the root. And the Pod boundary itself — why these containers share an IP
and reach each other on `localhost` — is a shared **network namespace**.

That's the whole shape of it. Everything above this layer in the book — the
scheduler deciding *where* a Pod lands, the controllers ensuring *how many* exist,
the networking wiring them together — eventually bottoms out in a node asking its
kernel to create a process with these three tricks applied. Kubernetes is the
distributed brain; namespaces, cgroups, and union filesystems are the hands.

Knowing this also makes you faster at the keyboard. `OOMKilled`? A cgroup memory
cap. A container that can't see a host process? A PID namespace doing its job.
Two containers fighting over port 80? They wanted separate network namespaces and
got the same one. The error messages stop being riddles once you know which trick
each one is about.
