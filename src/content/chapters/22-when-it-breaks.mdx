Here's the secret about Kubernetes troubleshooting: the cluster almost always
*already knows* what's wrong and is telling you. The skill isn't arcane
knowledge; it's the discipline to read the report in order instead of thrashing.
Four commands handle the overwhelming majority of incidents:

1. **`kubectl get pods`** — the wide shot. The `STATUS` and `RESTARTS` columns
   usually *name* the problem before you do anything else.
2. **`kubectl describe pod <name>`** — the detail view. Scroll to the bottom: the
   **Events** list is a timeline of what the scheduler, kubelet, and controllers
   tried, and why it failed.
3. **`kubectl logs <name>`** (add **`--previous`** for a container that already
   crashed) — the application's own account of its death.
4. **`kubectl get events --sort-by=.lastTimestamp`** — cluster-level events, for
   when the problem is bigger than one Pod (a node going `NotReady`, a failing
   image pull across the fleet).

Each step zooms in one level: from "which Pod is unhappy" to "what the platform
did about it" to "what the app said as it fell over." Resist the urge to skip to
step three and start editing code. The earlier steps usually tell you *which
layer* the problem lives in, which is half the answer.

</Section>

A Pod's `STATUS` is not a generic "broken." Each common failure state implicates a
specific **layer**, and knowing the map turns a vague alert into a directed
search.

- **`CrashLoopBackOff`** — the container starts, then exits, repeatedly, so the
  kubelet waits longer between each restart (the "back-off"). The fault is *inside
  the app*: an unhandled exception, a failed migration, a missing env var. Next
  stop: `logs --previous`.
- **`ImagePullBackOff`** / `ErrImagePull` — the kubelet can't *fetch* the image.
  Wrong tag, typo'd registry, or a missing pull secret. The app never ran at all;
  this is a packaging/registry problem, not a code one.
- **`Pending`** (unschedulable) — the **scheduler** can't place the Pod. `describe`
  spells out why: `insufficient cpu/memory`, or an unsatisfiable nodeSelector,
  affinity, or taint. Fix by lowering requests, adding capacity, or relaxing the
  constraint — *not* by touching the app.
- **`OOMKilled`** — the container exceeded its memory **limit** and the kernel's
  out-of-memory killer terminated it. Either the limit is too low or the app is
  leaking; raise the limit or fix the leak.

<Callout type="note">
Notice how the state alone narrows the layer: app logic (`CrashLoopBackOff`),
packaging (`ImagePullBackOff`), scheduling (`Pending`), or resources
(`OOMKilled`). You've eliminated three-quarters of the search space before reading
a single log line.
</Callout>

</Section>

Let's run the playbook for real. Below, a Pod named `web-1` is misbehaving. Start
wide with `kubectl get pods` and read the `STATUS`. Then `kubectl describe pod
web-1` to confirm the state and read the Events timeline. Finally, because the
container has already crashed, `kubectl logs web-1 --previous` to read what it
said on the way down. Run them in that order:

Watch how the picture sharpens. `get pods` tells you `web-1` is in
`CrashLoopBackOff` with five restarts — so the fault is *in the app*, and the
kubelet is dutifully backing off. `describe` confirms it: the container's last
state is `Terminated` with `Exit Code: 1`, and the Events show
`Back-off restarting failed container`. That's the symptom, not the cause. The
cause is in the **previous** container's logs:

```text
FATAL config: required env DATABASE_URL is not set
panic: missing DATABASE_URL
```

There it is — a missing environment variable, almost certainly an absent or
mis-keyed ConfigMap/Secret reference in the Pod spec. You would *never* have seen
that from `get pods` alone; the state told you *which layer*, and `--previous`
handed you the *exact* line. That is the entire method: let the state route you,
then read the evidence the cluster already wrote down.

</Section>

The four-command playbook assumes one sick Pod. Sometimes the pattern is
different: *many* Pods fail together, or a whole rollout stalls. When failure is
correlated, the cause usually lives one layer *below* the Pods, and a per-Pod
view will have you chasing symptoms.

Widen the lens. **`kubectl get nodes`** shows whether a machine has gone
`NotReady` — a dead kubelet or a failed node will take every Pod on it down with
it, producing a cluster-wide flurry of restarts and reschedules that looks like
chaos until you see the one bad node. **`kubectl get events --sort-by=.lastTimestamp -A`**
gives you the whole cluster's recent history in time order; a shared root cause —
a registry outage causing `ImagePullBackOff` everywhere, a `FailedScheduling`
storm because the cluster is genuinely full — jumps out as a repeated message.

<Callout type="tip">
Correlation is the clue. *One* Pod in `CrashLoopBackOff` is an app bug. *Twenty*
Pods across *five* nodes failing in the same minute is almost never twenty app
bugs — it's one shared dependency: a node, the registry, DNS, or the control
plane. Match the breadth of your investigation to the breadth of the failure.
</Callout>
