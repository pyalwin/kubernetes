Everything you've run so far could afford to forget. A web server's container can be
killed and recreated on another node, and nobody notices — it holds no state worth
keeping. That's because a container's filesystem is **ephemeral**: it's a thin
writable layer on top of the image, and it evaporates when the container goes away.
Reschedule the pod and it comes back blank.

For a database, an upload service, or anything that *remembers*, that's a
catastrophe. These workloads need storage that lives **independently of any pod** —
a real disk that exists before the pod, survives the pod, and can be reattached
wherever the pod is next scheduled. The whole design of Kubernetes storage follows
from that one requirement: separate the lifecycle of the data from the lifecycle of
the pod, completely.

The rest of this chapter is the machinery that makes that separation work — and,
pleasingly, it's the same declarative, pluggable, control-loop shape you've seen
everywhere else, pointed at disks.

</Section>

You just met this pattern with networking, and storage uses the same move. Kubernetes
refuses to hardcode knowledge of every storage system on earth — cloud block disks,
Ceph, NFS, on-prem SANs, niche vendor arrays. Instead it defines a standard contract,
the **CSI** (Container Storage Interface), and lets each vendor implement it.

A storage vendor ships a **CSI driver** — software that runs in your cluster and
knows how to do the vendor-specific work: create a volume, attach it to a node, mount
it into a pod, detach it, and delete it. Kubernetes speaks only to the CSI; the
driver translates those generic calls into AWS EBS API calls, or Ceph commands, or
whatever the backend needs.

<Callout type="note">
This is the *exact* shape of the CNI from the last chapter. Kubernetes defines an
interface and stays out of the implementation; vendors plug in. Adding support for a
new storage system means installing its CSI driver — no patch to Kubernetes itself.
The platform stays small while the ecosystem grows around it.
</Callout>

With a driver installed, the cluster can now *provision real storage on demand* — but
you don't call the driver directly. You ask for storage in Kubernetes' own terms, and
let a controller drive the CSI for you. Those terms are the next two objects.

</Section>

  quiz={{
    question: "What does a StorageClass enable when a PVC referencing it is created?",
    options: [
      { text: "Dynamic provisioning — the named provisioner creates a matching PV on demand.", correct: true },
      { text: "Faster image pulls for the pod." },
      { text: "Automatic encryption of etcd." },
    ],

>

Two objects model storage from opposite ends. A **PersistentVolume (PV)** is a piece
of real storage in the cluster — a specific disk, with a size and an access mode. A
**PersistentVolumeClaim (PVC)** is a *request* for storage by a pod: "I need 10Gi,
read-write-once." The pod mounts the PVC and never has to know which physical disk
backs it. That indirection is the point: the app author asks for capacity, not for a
particular volume.

Pre-creating PVs by hand, though, doesn't scale — and it drags storage details back
into the app author's lap. The fix is the **StorageClass**, and it's where the CSI
driver gets put to work. A StorageClass names a **provisioner** (a CSI driver) plus
parameters:

```yaml
kind: StorageClass
metadata: { name: fast-ssd }
provisioner: ebs.csi.aws.com
parameters: { type: gp3 }
reclaimPolicy: Delete
```

Now a PVC just references it:

```yaml
kind: PersistentVolumeClaim
spec:
  storageClassName: fast-ssd
  accessModes: [ReadWriteOnce]
  resources: { requests: { storage: 10Gi } }
```

This triggers **dynamic provisioning**. A controller sees an unbound PVC asking for
`fast-ssd`, calls that class's provisioner (the EBS CSI driver), which creates a real
gp3 disk and produces a PV representing it. Kubernetes then **binds** the PVC to that
PV, and the pod mounts it. No admin pre-provisioned anything; no one waited. Here's
that flow:

    options: [
      { text: "The underlying volume is deleted along with the claim.", correct: true },
      { text: "Nothing — the disk is always preserved." },
      { text: "It is automatically snapshotted to a new PV." },
    ],
  }}
>

There's one decision left, and it's the one that bites people: when a PVC is deleted
and its PV is released, what happens to the **real disk** underneath? That's set by
the **reclaim policy**.

- **Delete** — the provisioner destroys the backing volume too. The CSI driver tells
  the storage backend to delete the actual disk, and the data is gone. This is the
  sensible default for scratch space, caches, and dev environments, where keeping
  orphaned disks just wastes money.
- **Retain** — the PV (and its data) is *kept* after the claim disappears. The volume
  moves to a `Released` state but is not reused or erased. An admin can later inspect
  it, recover the data, archive it, and clean it up manually.

The difference is the difference between "convenient" and "safe." A `Delete` policy
on your production database means a single fat-fingered `kubectl delete pvc` can
destroy the disk along with the claim — irreversibly.

<Callout type="tip">
For anything you'd be sad to lose, set **Retain**. It costs you the chore of cleaning
up released volumes by hand, and buys you a recoverable copy when someone deletes the
wrong thing. That's a trade worth making for stateful data every time.
</Callout>

Step back and the whole chapter is one idea: data's lifecycle is decoupled from the
pod's. The CSI lets any backend plug in; the StorageClass provisions on demand; the
PVC→PV binding connects an abstract request to a concrete disk; and the reclaim
policy decides the disk's fate when the claim is gone. A pod can come and go freely
because none of its durable state lives inside it.

</Section>
