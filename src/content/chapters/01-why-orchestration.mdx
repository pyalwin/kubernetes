>

A **container** is the thing Kubernetes spends its entire life managing, so we
need a one-paragraph picture of it before anything else — and one paragraph is
genuinely all it takes to start. A container is a single program packaged
together with everything it needs to run — its code, its libraries, its files —
sealed into one isolated, portable bundle that behaves the same on your laptop,
in a test pipeline, or on a server in a datacenter. You start one with a single
command; something like `docker run nginx` fetches a packaged web server and
runs it. That sentence is the entire amount of container knowledge this chapter
needs.

<Callout type="note" title="New to containers? You're in the right place">
You need **no** prior Kubernetes experience, and only the rough picture above —
*"a container is a packaged, isolated program."* We build everything else from
scratch as we go. A little comfort at a command line helps; nothing more. When
you're curious what's actually inside that "bundle," Chapter 2, *What a Container
Actually Is*, takes one apart and shows there's no magic in it — just a Linux
process wearing a disguise.
</Callout>

So: running *one* container is easy. One command, and a web server is live; it
feels like the hard part is over. It isn't — running one container is easy in the
way lighting one candle is easy. The hard part is keeping a thousand candles lit,
in the wind, while someone periodically walks through the room knocking them
over — and never being the person who has to notice and relight each one.

So let's not start with a definition of Kubernetes. Definitions of Kubernetes
are uniformly terrible; they use the word "orchestration" to explain the word
"orchestration." Let's instead try to run a real service the manual way, take on
each chore as it appears, and feel exactly where it stops being possible. The
features of Kubernetes will then arrive not as vocabulary to memorize but as the
obvious answer to a problem you just felt.

There is one idea waiting at the end of this. You may already know its name.
Hold on for it.

</Section>

>

Ship your container to one server. Day one is fine. Then the failures begin, and
they are not exotic — they are Tuesday.

- **The process crashes.** A bug, an out-of-memory kill, a bad deploy. Your
  service is down until *someone notices* and runs the container again. So you
  write a little shell loop that restarts it. Fine.
- **The server reboots.** Kernel update, hardware blip. Your restart loop reboots
  with it and is gone. So you make it a `systemd` unit. Fine.
- **The whole machine dies.** Now `systemd` can't help — it died too. You need
  the container to come back *on a different machine*. Which machine? Is there
  room? Who decides?
- **Traffic triples.** One copy isn't enough. You start three, on three boxes,
  and now something has to spread requests across them and stop sending traffic
  to the one that just crashed.
- **You deploy v2.** You'd like to replace the three copies *without* a window
  where the site is down — a few at a time, rolling back instantly if v2 is bad.

Notice the shape. Every single item is the same move: *look at what is true,
compare it to what should be true, and take an action to close the gap.* The
process should be running — it isn't — start it. Three copies should exist — two
do — start one. You have been hand-cranking a control loop, badly, because you
have to sleep.

<Callout type="note">
The work here scales with **machines × services**. Ten services on twenty nodes
is two hundred little loops, each with its own failure modes, all depending on a
human being awake. This is the wall everyone hits, and it is the reason
orchestrators exist.
</Callout>

</Section>

<Section
  id="desired-state"
  title="Declaring the outcome"
  brief={{
    problem: "Step-by-step instructions are fragile: if a step fails halfway, the system is left in an unknown state with no obvious recovery.",
    mechanism: "Declare the desired end state instead. There is no 'halfway' to a description — only a gap to be closed.",
    takeaway: "Declarative desired state is what makes automated recovery tractable.",
  }}
  quiz={{
    question: "Why is a declarative 'desired state' more robust than an imperative script in a system where anything can fail at any moment?",
    options: [
      { text: "Declarative configs are written in YAML, which is faster to parse." },
      { text: "A description has no half-finished state — the system can always just compare and act to close the gap, however it got knocked off course.", correct: true },
      { text: "Imperative scripts cannot express what you want." },
    ],
  }}
>

Here is the pivot that makes automation possible. Stop writing *instructions* and
start writing a *description of the desired result*.

Imperative, the way you'd script it by hand:

```bash
# do this, then this, then this...
start_container web on node-3
if traffic_high: start_container web on node-4
if node-3 down: start_container web somewhere else
```

The trouble: what happens if the script dies on line two? Now the world is in a
state your script never anticipated, and "run it again" might start a fourth
copy or none. Imperative steps assume nothing fails. In a fleet of machines,
*something is always failing.*

Declarative, the way Kubernetes wants it:

```yaml
# this is simply true. make it so, and keep it so.
kind: Deployment
spec:
  replicas: 3
  image: web:v2
```

This says nothing about *how*. It is a statement of fact about the world you
want: three copies of `web:v2`, always. There is no "halfway" to that sentence.
Whatever happens — a crash, a dead node, a botched deploy — the system's job is
unchanged: look at reality, compare it to this, and act. That is a job a machine
can do tirelessly, and it is the whole reason the next idea works.

</Section>

<Section
  id="control-loop"
  title="The control loop"
  brief={{
    problem: "A declared desired state is useless unless something continuously enforces it.",
    mechanism: "A control loop observes actual state, compares it to desired, acts to reduce the difference, and repeats forever.",
    cost: "Enforcement is never 'done' — it costs continuous watching and reconciliation work.",
    takeaway: "Self-healing is not a feature bolted on; it is what a control loop does by definition.",
  }}
  quiz={{
    question: "You delete a Pod that a Deployment manages. Moments later an identical one appears. Why?",
    options: [
      { text: "Kubernetes logged an error and gave up." },
      { text: "A controller compared actual (2 Pods) to desired (3 Pods) and created one to close the gap.", correct: true },
      { text: "Deleting Pods is not allowed." },
    ],
  }}
>

The mechanism that enforces desired state is the **control loop**, and it is so
simple it feels like it shouldn't work:

1. **Observe** the actual state of the world.
2. **Compare** it to the desired state you declared.
3. **Act** to reduce the difference.
4. Go to 1. Forever.

That's it. A piece of software that runs this loop for one kind of thing is
called a **controller**. The controller responsible for your three replicas does
nothing but ask, over and over, "are there three healthy copies?" When the
answer is no — a Pod crashed, a node vanished — it creates one. You wrote no
recovery code. You declared an outcome, and a loop maintains it.

This reframes a phrase you'll hear constantly. Kubernetes "self-heals" not
because it is intelligent, but because *healing is just the loop noticing a gap
and closing it.* Kill a Pod and watch it return: you are not seeing magic, you
are seeing step 3.

<Callout type="tip">
Hold onto this loop. Almost every component you meet for the rest of this book —
the thing managing replicas, the thing managing nodes, the thing wiring up
networking — is another instance of exactly this four-step loop, pointed at a
different slice of the world.
</Callout>

</Section>

If a dozen control loops are all enforcing desired state, they need two things:
somewhere to read what *desired* means, and somewhere to record what they
*observe*. That shared brain is the **control plane**.

You'll meet every part of it in Chapter 5. For now, just the spine — click each
piece:

<Figure number="1.1" caption="The two pieces every control loop depends on. The API server is the only door into the cluster; etcd is the cluster's memory.">
  <Diagram client:visible id="cp-intro" title="The control plane, minimal" nodes={controlPlaneNodes} />
</Figure>

The **API server** is the single front door. You don't edit cluster state
directly, and neither do the controllers — *everyone* goes through the API
server. That one rule is why the system is observable and secure: there is one
place where every change is validated, authorized, and recorded.

Behind it, **etcd** is the cluster's memory: a consistent, replicated key-value
store holding both halves of the picture — what you declared, and what currently
exists. When a controller "compares desired to actual," this is what it reads.
When a node dies and its Pods need rescheduling, the gap becomes visible *here*,
and some loop springs into action.

</Section>

`kubectl` is the tool you'll spend the rest of this book in. It is not special —
it is just a well-mannered client that makes requests to the API server, exactly
as the controllers do. Asking the cluster what it's running is the most basic
thing you can do, so start there.

This terminal is simulated: it returns the real shape of `kubectl` output without
needing a cluster. Try `kubectl get nodes` and `kubectl get pods`.

  <KubectlTerminal client:visible id="ch1-term" commands={firstPodsTerminal} />

Each line came back through the API server reading etcd. You didn't reach a
single node directly. That indirection — one front door, one source of truth,
many loops reconciling against it — *is* Kubernetes. The rest is detail, and the
detail is where it gets good.

The framing in this chapter — declarative desired state enforced by
reconciling controllers — is the foundation the official documentation builds
everything else on.

- **Kubernetes Documentation — *What is Kubernetes?* and *Kubernetes
  Components*.** The canonical description of the control plane, the API server,
  and etcd. `kubernetes.io/docs/concepts/overview/`

  it.

  (ACM Queue, 2016).** How a decade of running control loops at Google became
  Kubernetes — the lineage of the reconciliation model.
