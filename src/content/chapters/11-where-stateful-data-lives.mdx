Every virtue we've built up has a dark side for one kind of app. Pods are
disposable; controllers replace them freely; the scheduler moves work between
nodes. Wonderful for a stateless web server — catastrophic for a database.

The root of the trouble is how a container's filesystem works. Recall from early
on that an image is a stack of *read-only* layers. When a container runs, it gets
one thin **writable layer** on top, and anything it writes goes there. That
writable layer is born with the container and **dies with it**. Restart the
container — a crash, an OOM kill, a rolling update — and you get a *fresh* writable
layer. Whatever the old one held is discarded.

For a stateless app this is a feature: every Pod starts clean and identical. But a
database's entire job is to *remember*. If its files live in that writable layer,
then the first crash, the first node failure, the first deploy wipes the database.
The very disposability that makes Pods reliable makes them, by default, amnesiac.

So we need storage that is *not* part of the container — storage with a lifetime
of its own, that a Pod can attach to and that survives the Pod being restarted,
rescheduled, or replaced. Kubernetes builds that up in layers, starting small with
the Volume and ending with cluster-wide persistent storage. We'll climb that
ladder one rung at a time.

</Section>

The first rung is the **Volume**. A Volume is storage declared on the *Pod* and
mounted into one or more of its containers. Because it belongs to the Pod rather
than to a single container's writable layer, it solves two problems at once.

First, it survives **container restarts**. If a container crashes and the kubelet
restarts it within the same Pod, a Volume mounted into it keeps its contents. The
ephemeral writable layer resets; the Volume doesn't.

Second, it enables **sharing between containers** in a Pod — the mechanism we
waved at back in Chapter 7. The app writes logs to a Volume mounted at
`/var/log/app`; the sidecar mounts the same Volume and reads them. That shared
patch of filesystem is a Volume.

There are many Volume types. `emptyDir` is scratch space created empty when the
Pod starts — great for caches and inter-container handoff. The `configMap` and
`secret` mounts from Chapter 10 are Volume types too. But here's the ceiling: a
plain Volume's lifetime is tied to the **Pod**. An `emptyDir` is created with the
Pod and **deleted with it**. So a Volume survives a *container* restart but not the
*Pod* being deleted and rescheduled — which is exactly the event a database must
survive. We've climbed one rung and we're still not high enough.

</Section>

<Section

      { text: "A PV is a real piece of cluster storage that exists independently of any Pod; a PVC is a user's request that Kubernetes binds to a matching PV.", correct: true },

The top rung introduces storage with a life entirely its own, and it does so by
**splitting the concept in two** — separating who *provides* storage from who
*needs* it.

A **PersistentVolume (PV)** is a real piece of storage in the cluster: an AWS EBS
disk, a GCE persistent disk, an NFS export, a slice of a storage system. It's a
cluster-level object that exists *independently of any Pod*. An administrator can
create PVs ahead of time, or a **StorageClass** can provision them dynamically on
demand. Either way, a PV represents actual, durable capacity that doesn't vanish
when a Pod does.

A **PersistentVolumeClaim (PVC)** is a *request* for storage, written by the
application developer: "I need 10Gi that I can read and write." The PVC says what
the app *needs* — size, access mode — without naming a specific disk. The Pod
references the **claim**, never the PV directly. Click through the chain:

This is the same indirection you've now seen three times. A Service lets callers
name a stable identity instead of a Pod IP; a Deployment lets you declare a count
instead of managing Pods; a PVC lets an app *request* storage instead of knowing
where it physically lives. The developer asks for "10Gi, read-write"; whether
that's backed by EBS in prod or a local disk on your laptop is somebody else's
concern. Decoupling by request is one of the deepest patterns in the whole system.

</Section>

<Section
  id="access-modes-bind"
  title="Access modes and the bind"
  brief={{
    problem: "Not all storage can be used the same way — some can be written by one node, some read by many — and a claim has to be matched to a volume that can satisfy it.",
    mechanism: "A PVC specifies an access mode (RWO/ROX/RWX); Kubernetes binds it to a PV that offers at least that capability, then the Pod mounts the claim.",
    takeaway: "The access mode is the compatibility contract; the bind is Kubernetes pairing a claim with a volume that honors it.",
  }}
  quiz={{
    question: "A PVC requests ReadWriteOnce (RWO). What does that tell Kubernetes about how the storage may be used?",
    options: [
      { text: "Any number of nodes may read and write it simultaneously." },
      { text: "It can be mounted read-write by a single node at a time — so it must bind to a PV that supports RWO.", correct: true },
      { text: "It can only ever be read, never written." },
    ],
  }}
>

When you request storage you also state *how* it needs to be mounted, because
storage backends genuinely differ in what they allow. This is the **access mode**,
and it's the compatibility contract between a claim and a volume:

- **ReadWriteOnce (RWO)** — mounted read-write by a single node at a time. The
  most common mode; it fits block storage like cloud disks, and it's what a typical
  single-instance database wants.
- **ReadOnlyMany (ROX)** — mounted read-only by many nodes at once. Good for
  shared static assets.
- **ReadWriteMany (RWX)** — mounted read-write by many nodes simultaneously. Only
  some backends (like NFS) support it, because concurrent multi-writer access is
  genuinely hard.

The **bind** is the moment Kubernetes pairs your claim with a volume that can
honor it. You create a PVC asking for "10Gi, RWO"; the control plane finds an
existing PV that's at least that big and supports RWO (or dynamically provisions
one via the StorageClass), and binds them together one-to-one. From that point the
PVC is `Bound`, and your Pod mounts it like any volume. Watch the pairing:

<Callout type="tip">
The payoff is what happens on reschedule. If the Pod's node dies and the Pod is
recreated elsewhere, the storage follows the *claim*, not the node — the new Pod
re-mounts the same PV and the data is intact. That is durability surviving exactly
the churn Pods were designed to endure.
</Callout>

The storage lives on, bound to a claim, while Pods come and go above it. With that
in place, Part II has given every workload what it needs: identity, replication, a
stable address, configuration, and now durable data. The last chapter zooms back
out to the cluster as a shared resource — how many teams safely split one cluster
without trampling each other.

</Section>
